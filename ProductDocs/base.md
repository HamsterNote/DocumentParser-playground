# 自主权声明

在阅读本篇文档时，代表你对当前项目内的事情具有完全的自主权，你可以自由安装和移除 NPM 包，自由地修改当前项目内的文件，包括 ProductDocs 里面的文档，我允许你不计成本（指你的上下文及 token）实现功能，token 有的是，随你挥霍，有时可能因为需求过大，允许你对单个需求文档进行分批次实现

# 拿不定主意时

如果你有发现拿不定主意的情况时，可以中断像我提问，我会回答你的问题

# 核心宗旨

我要求你作为一个可以自主的工作人员，你阅读我的产品文档，并逐条实现功能，在实现功能之前， 从 dev 分支新建一个分支，在每次实现并测试完成后，将产品文档中对应的一条功能标记为完成（一般是指 Markdown 上的 Checkbox 设为勾选），在 CHANGELOG 中标记你完成的内容，并将代码提交 Pull Request 到 dev 分支，至此结束你的单次 session，我要求你的工作流程如下：

1. 如果有上次未完成的任务，则继续进行上次的工作
2. 如果没有未完成的任务，则在产品文档中按顺序找到还没有做的一条产品需求，将此产品需求作为你本次工作的目标
3. 阅读产品文档，生成临时的计划，在项目根目录下的 plan 文件夹中生成跟你当前工作相关的内容为文件名.md，且永远不要删除这个文件
4. 你的计划每一条是一个 TODO 事项，在完成一条计划时就勾选它标记为完成，完成后继续下一条计划
5. 在完成计划，或者中途需要测试时，尽可能生成比较完善的单元测试代码，并测试通过为止，在需要集成测试时，你可以使用 playwright 来测试浏览器，并将测试结果反馈在上下文中，修复问题，直到没有问题为止。在测试过程中，不要发现测试结果错误而盲目地修改预期值，而是要自主判断是代码问题，还是预期值问题，再做相应修改
6. 在完成代码后，需要进行检测，是否符合 eslint、prettier、sonarjs 要求，这三个都配置在 eslint 中了，所以你只需要检查是否有 eslint error 就可以了，如果有问题，你要自己把它们修复
7. 当完成了代码、测试，自己写一个 commit message，格式：“feat: 增加 XX 功能”，并 push 到你新建的分支，然后提一个 pull request 到 dev 分支
